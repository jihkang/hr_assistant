# Backend Detailed Design

## 1. Goal

Build a backend for an internal HR and General Affairs assistant with:

- email/password login
- rank- and department-aware authorization
- asset search and rental workflows
- HR policy Q&A
- LLM + RAG orchestration
- traceable evaluation and quality improvement loops

The backend stack is:

- FastAPI
- PostgreSQL
- pgvector
- LangGraph
- LangSmith
- OpenAI API

## 2. Domain Rules

### Departments

- `인사`
- `개발`
- `마케팅`
- `재무`
- `영업`
- `운영`

### Ranks

- `사장`
- `부사장`
- `부서장`
- `부장`
- `과장`
- `대리`
- `사원`

### Admin rule

Admin access is derived from department and rank.

- If `department = 인사`, admin is allowed when rank is `과장` or higher.
- For all other departments, admin is allowed when rank is `부장` or higher.

This rule should be enforced in the backend service layer and route guard layer. It should not depend on the LLM.

## 3. Authentication Design

### Scope

- Signup is not supported.
- Users are pre-created by the organization or seed/admin tooling.
- Login uses `email + password`.

### Recommended session model

Use token-based auth with cookies.

- `access token`: short-lived JWT
- `refresh token`: longer-lived JWT or opaque token
- both stored in `httpOnly`, `secure`, `sameSite=lax` cookies

### Required auth endpoints

- `POST /api/v1/auth/login`
- `POST /api/v1/auth/logout`
- `POST /api/v1/auth/refresh`
- `GET /api/v1/auth/me`

### Login flow

1. User submits email and password.
2. Backend validates password hash.
3. Backend computes current admin eligibility from `department` and `rank`.
4. Backend issues cookies.
5. Frontend calls `/auth/me` to hydrate the current user session.

## 4. Core Data Model

This section follows the schema proposal and fixes the intent of each table.

### `users`

Stores the current employee state.

Main fields:

- `name`
- `email`
- `password_hash`
- `department`
- `rank`
- `hire_date`
- `annual_leave_total`
- `annual_leave_used`
- `annual_leave_remaining` generated by PostgreSQL
- `employment_status`
- `is_active`
- `last_login_at`
- `created_at`
- `updated_at`

Notes:

- `annual_leave_remaining` must be generated in the database from:
  `annual_leave_total - annual_leave_used`
- admin is derived, not stored as the source of truth

### `user_rank_history`

Stores promotion, demotion, and correction events.

Fields:

- `user_id`
- `previous_rank`
- `new_rank`
- `change_type`
- `reason`
- `changed_at`

Purpose:

- auditability
- explain why admin eligibility changed
- preserve historical rank transitions

### `assets`

Stores rentable or trackable company assets.

Suggested fields:

- `name`
- `category`
- `serial_number`
- `status`
- `location`
- `owner_department`
- `requires_approval`
- `created_at`
- `updated_at`

Notes:

- `status` should eventually move to a constrained enum or check
- `requires_approval` helps the assistant explain approval paths

### `asset_rentals`

Stores the rental lifecycle.

Suggested fields:

- `asset_id`
- `user_id`
- `requested_at`
- `approved_by`
- `approved_at`
- `returned_at`
- `status`
- `note`

Status values:

- `requested`
- `approved`
- `denied`
- `returned`

### `policy_documents`

Primary RAG source table for HR and GA policy content.

Suggested fields:

- `title`
- `document_type`
- `source`
- `department`
- `visibility`
- `effective_date`
- `content`
- `vector`
- `created_at`

Purpose:

- retrieve policy evidence for answers
- apply metadata filters before retrieval
- track current and historical policy versions

### `chat_logs`

Stores user-facing conversation logs.

Suggested fields:

- `user_id`
- `prompt`
- `response`
- `citations`
- `metadata`
- `langsmith_trace_id`
- `created_at`

Purpose:

- auditing
- quality review
- replaying low-quality sessions

### `langsmith_traces`

Optional persistence layer for trace references or snapshots.

Suggested usage:

- store the external LangSmith trace id
- optionally store selected trace payloads for internal audit

This table should not become the primary observability platform. LangSmith remains the source of truth for full trace inspection.

## 6. Authorization Model

### Non-negotiable rule

Authorization is enforced before any LLM call when the request touches:

- employee-specific data
- admin pages
- restricted policies
- approval workflows

### Checks

- `is_authenticated`
- `is_active`
- `employment_status == active` for normal flows unless explicitly allowed
- `is_admin_by_rule`

### Admin derivation

Pseudo rule:

```text
if department == "인사":
    admin if rank in [사장, 부사장, 부서장, 부장, 과장]
else:
    admin if rank in [사장, 부사장, 부서장, 부장]
```

### Why this matters

This rule is also a guardrail test target:

- HR manager should pass
- HR staff should fail
- non-HR department manager should pass only at `부장` and above
- non-HR `과장` should fail

## 7. Assistant Architecture

### Recommended pattern

Use a controlled agent workflow, not an open-ended autonomous multi-agent design.

Reason:

- HR and GA are policy-heavy domains
- permission boundaries must be deterministic
- auditability matters more than agent creativity

### LangGraph responsibilities

LangGraph should orchestrate:

1. request classification
2. auth and permission context loading
3. tool selection
4. RAG retrieval with metadata filters
5. answer drafting
6. citation packaging
7. guardrail validation
8. logging and evaluation hooks

### Nodes

Suggested initial graph nodes:

- `load_user_context`
- `classify_request`
- `check_permission`
- `retrieve_documents`
- `query_business_data`
- `generate_answer`
- `validate_answer`
- `persist_logs`

## 8. RAG Design

### Source types

Initial sources:

- policy markdown or seeded document text
- HR/GA procedure documents
- asset policy documents

### Retrieval rules

Before retrieval:

- derive user permission context
- determine relevant domain: HR, asset, admin, general policy
- apply metadata filters: `department`, `visibility`, `effective_date`

After retrieval:

- drop documents outside user visibility
- prefer newer documents when policy conflicts exist
- require citations in the final response

### Vector strategy

Use `pgvector` in PostgreSQL.

Recommended approach:

- one embedding vector per chunk
- separate chunk table later if document volume grows

For MVP, a single `policy_documents` table is acceptable. For scale, split into:

- `policy_documents`
- `policy_document_chunks`

## 9. Guardrail Design

Guardrails should be layered.

### Layer 1: route and permission guardrails

- reject unauthorized admin access before LLM execution
- reject access to restricted employee or policy data

### Layer 2: tool guardrails

- restrict write tools to admin-only flows
- allow read tools only within permitted scope

### Layer 3: retrieval guardrails

- metadata-filter documents before they reach the model
- do not pass restricted documents into prompt context

### Layer 4: output guardrails

- require citations for policy answers
- refuse unsupported claims if no evidence exists
- mask sensitive values

### Layer 5: evaluation guardrails

- compare answer quality over time
- detect regressions in citation quality, permission safety, and hallucination rate

## 10. LangSmith Design

LangSmith is used for:

- trace inspection
- prompt comparison
- experiment tracking
- regression checks
- dataset-based evaluation

### What to send to LangSmith

- user intent classification result
- selected graph path
- retrieved document ids
- tool calls
- final answer
- guardrail outcomes

### What to store locally

- `langsmith_trace_id`
- selected evaluation summaries
- references from chat logs to traces

Do not duplicate every full trace payload in PostgreSQL unless audit requirements demand it.

## 11. Quality Loop

### Offline evaluation

Build a seed evaluation dataset with:

- login/auth scenarios
- admin access scenarios
- asset search scenarios
- rental policy questions
- leave policy questions
- prompt injection attempts

Key metrics:

- answer correctness
- citation presence
- citation relevance
- permission violation rate
- refusal correctness

### Online review

Sample real chat logs and compare:

- model version
- prompt version
- graph version
- retrieval version

### Improvement cycle

1. detect failures from logs and evals
2. inspect LangSmith traces
3. update prompt, retrieval logic, or tool rules
4. rerun evals
5. release only if regression thresholds pass
